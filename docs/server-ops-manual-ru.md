# Руководство по эксплуатации сервера связи (Go)

## 1. Назначение

`itmcomd` — это сервер связи для телеметрии и команд управления.
По схеме он соответствует роли **«ПО сервера связи»** и работает как шлюз между:

- контроллером (`Motorola ACE 3600`, TCP/RS-232),
- сервером данных/интеграцией (`PI Moto Интерфейс`),
- внутренними API-клиентами и БД.

Ключевые функции:

- прием и разбор протоколов `Legacy` и `ATSWP`,
- маршрутизация сообщений (TCP, очереди, COM),
- выдача совместимого API (`/v1/compat/*`),
- сохранение событий/данных в MSSQL (опционально),
- мониторинг состояния и метрик.

## 2. Компоненты и зоны ответственности

### 2.1 Контроллер (поле)

- источник телеметрии;
- получатель команд;
- каналы связи: `TCP` и/или `RS-232`.

### 2.2 Сервер связи (`itmcomd`)

- держит TCP listeners для модемных сессий;
- управляет жизненным циклом соединений и watchdog;
- парсит пакеты, ведет очереди `command/config/commstate/monitor`;
- обеспечивает API для чтения состояния и отправки команд.

### 2.3 Сервер данных/интеграция

- получает обработанные данные;
- инициирует команды в обратном направлении через API;
- использует данные и события для верхнего уровня.

### 2.4 База данных MSSQL (опционально)

- runtime-конфигурация (`listeners`, `routing`, `comPorts`, `comRoutes`);
- журналы/события/статистика;
- поддержка legacy-режима с совместимостью stored procedures.

## 3. Сетевые и интерфейсные точки

- Модемные входы: `listeners[]` из `config.json` (например `0.0.0.0:8010`).
- HTTP API: определяется `httpPort` в конфиге.
- COM-интерфейсы: `comPorts` + `comRoutes`.
- Внешняя публикация через firewall/NAT выполняется на стороне инфраструктуры.

## 4. Предварительные требования

- Go toolchain (версия из `go.mod`).
- Доступ к MSSQL (если используется БД режим).
- Доступ к COM-портам ОС (если используется RS-232).
- Открытые сетевые порты для:
  - входящих модемных TCP сессий,
  - HTTP API сервиса,
  - исходящего/входящего MSSQL.

## 5. Конфигурация

Базовый файл: `config.json`.

Критичные параметры:

- `listeners`: входящие TCP точки приема.
- `routing`: соответствие `siteID -> modemID`.
- `comPorts` / `comRoutes`: мост между логикой сервиса и COM.
- `dbConnString`: строка подключения к MSSQL.
- `dbAutoMigrate`: авто-миграция служебных таблиц.
- `dbUseStoredProcedures`: приоритет legacy SP совместимости.
- `apiToken`: Bearer-токен защиты API (пустой = без токена).
- `apiRateLimitPerMinute`: лимит запросов на IP.
- `statisticsIntervalSeconds`, `statisticsRetentionHours`: статистика/ретеншн.

## 6. Запуск и остановка

### 6.1 Локальный запуск

```bash
make run
```

Альтернативный alias:

```bash
make alesklesik1990
```

### 6.2 Сборка бинарника

```bash
make build
./itmcomd -config ./config.json
```

### 6.3 Остановка

- штатно: `SIGINT`/`SIGTERM`;
- сервис выполняет graceful shutdown и закрывает ресурсы.

Подробности жизненного цикла: `docs/service-lifecycle.md`.

## 7. API для эксплуатации

Основные группы endpoint'ов:

- runtime управление:  
  - `GET/PUT /v1/runtime`  
  - `GET/POST/DELETE /v1/runtime/listeners`  
  - `GET/POST/DELETE /v1/runtime/routing`  
  - `GET/POST/DELETE /v1/runtime/comports`  
  - `GET/POST/DELETE /v1/runtime/comroutes`
- мониторинг:  
  - `GET /v1/modems`  
  - `GET /v1/metrics`
- compat-операции:  
  - `POST /v1/compat/connect`  
  - `POST /v1/compat/disconnect`  
  - `POST /v1/compat/break`  
  - `POST /v1/compat/docommand`  
  - `GET /v1/compat/getdataset`  
  - `GET/POST /v1/compat/get|setuse*`  
  - `GET /v1/compat/get*data`

Для compat API действует модель владения клиентом (`machineName`) для защиты от конфликтующих операторов.

## 8. Типовые сценарии эксплуатации

### 8.1 Ввод в работу

1. Заполнить `config.json`.
2. Проверить соединение с MSSQL (если включен DB режим).
3. Запустить `make run`.
4. Убедиться, что listener'ы поднялись.
5. Проверить `GET /v1/modems` и `GET /v1/metrics`.

### 8.2 Отправка команды контроллеру

1. Захватить владение (compat connect с `machineName`).
2. Выполнить `POST /v1/compat/docommand`.
3. Проверить очереди/состояние через compat/data endpoints.
4. Освободить владение (compat disconnect).

### 8.3 Изменение runtime без рестарта

1. Применить изменения через `/v1/runtime*`.
2. Проверить обновленный runtime через `GET /v1/runtime`.
3. Проверить факт применения по метрикам/списку модемов.

## 9. Мониторинг и регулярные проверки

Ежесменный минимум:

- доступность TCP listener'ов;
- `GET /v1/metrics` без резких всплесков `parseErrors/readErrors/writeErrors`;
- размер и активность очередей;
- успешность записи в БД (если включена);
- статус COM каналов и количество reopen/retry.

Рекомендуется накапливать историю метрик и настроить алерты на:

- рост malformed/parse ошибок;
- частые reconnect/disconnect;
- деградацию времени ответа API;
- постоянные ошибки записи в БД.

## 10. Аварийные ситуации и действия

### 10.1 Нет связи с контроллером

- проверить firewall/NAT и доступность TCP порта;
- проверить правильность `listeners` и маршрут до контроллера;
- при RS-232 проверить доступность COM порта и параметры.

### 10.2 Ошибки БД

- проверить `dbConnString`, доступность SQL и права пользователя;
- временно можно продолжить в in-memory режиме, но без гарантированной персистентности;
- после восстановления БД проверить отложенные/текущие записи.

### 10.3 Рост битых пакетов

- проверить качество канала связи;
- сверить настройки и версию протокола на стороне контроллера;
- проанализировать parser-ошибки и packet boundaries.

## 11. Обновление и откат

### 11.1 Обновление

1. На новой ветке выполнить изменения.
2. Прогнать `make preprod`.
3. Выпустить PR, дождаться зеленого CI.
4. Выполнить deployment по регламенту площадки.

### 11.2 Откат

- вернуться на предыдущий стабильный бинарник/релизный tag;
- восстановить предыдущий `config.json` при необходимости;
- проверить работоспособность по чек-листу раздела 9.

## 12. Эксплуатационный чек-лист перед сменой

- [ ] Сервис запущен, API отвечает.
- [ ] TCP listener'ы доступны.
- [ ] Контроллеры в онлайне (`/v1/modems`).
- [ ] Критичных ошибок в метриках нет.
- [ ] БД доступна (если включена).
- [ ] COM маршруты работают (если используются).
